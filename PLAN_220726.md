## Phase 1 — Smoke test the plumbing (cheap, catches most setup bugs)

**BUILD (Claude Code):** create `scripts/smoke_test.py` that:
- instantiates `from model.pacl import open_clip_pacl` (this triggers the ~600MB open_clip
  `laion2b-s34b-b88K` weight download the first time),
- runs a forward pass on a **random** image tensor `[2, 3, 400, 400]` and a tokenized caption,
- asserts the shapes: `forward_visual → [2, 625, 512]`, `forward_text → [2, 512]`,
  `patch_alignment → [2, 625]`, and `model(images, caps)` returns two `[2, 512]` tensors,
- runs one `ClipLoss` forward and prints the scalar loss,
- runs on GPU if available, else CPU (CPU is fine for this test),
- prints "SMOKE TEST PASSED" at the end.

Then run:
```bash
python scripts/smoke_test.py
```

**Success criterion:** prints "SMOKE TEST PASSED" with the expected shapes and a finite loss.
This confirms the model builds, weights download, spaCy/tokenizer load, and shapes are correct —
*without needing any data*.

## Phase 3 — Short training run (proves the objective learns)

Because the CLIP backbone is frozen and already semantically coherent, only the small projection
heads train — so a few epochs on a small subset is enough to see localization emerge. Single GPU:

```bash
python train_pacl.py \
  --train_dir  /path/to/MSCOCO/val2017/ \
  --train_anno /path/to/MSCOCO/annotations/captions_val2017.json \
  --val_dir    /path/to/MSCOCO/val2017/ \
  --val_anno   /path/to/MSCOCO/annotations/captions_val2017.json \
  --batch_size 64 --epochs 5 --lr 1e-4 --amp \
  --save_path pacl_ft.pth
```

Notes:
- `--batch_size` is the **global** batch and sets the number of in-batch contrastive negatives.
  64 is a *sanity* value (far below the paper's 1024) — expect lower quality, not a reproduction.
- Lower `--batch_size` further only if you hit CUDA OOM.

**Success criterion:** `train_loss` **decreases** over steps/epochs and `pacl_ft.pth` is written.
A falling contrastive loss is the first real evidence the alignment is being learned.

## Phase 4 — Inference — the "does it actually work?" moment

```bash
python pacl_inference.py --weights pacl_ft.pth \
  --image "<url-or-local-path-of-an-image-with-a-cat>" \
  --caption "a picture of a cat." \
  --out results/activation_map.png
```

**Success criterion (qualitative):**
- The activation map in `results/activation_map.png` is **bright over the cat and dark elsewhere**.
- Changing the caption (`"a dog."`, `"grass."`, `"a car."`) **moves the hot region** to the
  matching object / goes dark when the concept is absent.

If the map is uniform noise: train longer / on more data, and double-check `--weights` points at
the checkpoint you just trained. If it localizes → **PACL works end-to-end.** 🎉

## Phase 5 — Prove it works *well* (quantitative mIoU) + scale up

This is the biggest gap in the repo today. Two parts:

### 5a. BUILD (Claude Code): `evaluate_miou.py`
An evaluation harness that reports the paper's metric. It should:
- take a segmentation dataset (start with **Pascal VOC** — 20 foreground classes + background)
  with its validation images and ground-truth masks,
- for each image: run `forward_visual`, and for each class name build a text prompt, run
  `forward_text` + `patch_alignment` to get a per-patch score map, stack across classes,
- upsample the 25×25 (or finer) score maps to image resolution, `argmax` over classes per pixel
  (with a background threshold), producing a predicted segmentation,
- compute **mean IoU** against ground truth over all classes; print per-class IoU and mIoU.
- Optional fidelity improvements to add: the paper's **stride-4 inference trick** (finer patches)
  and a **`softmax`** weighting variant in `patch_alignment` (vs the current `sigmoid(10·s)`).

**Success criterion:** produces a numeric mIoU on Pascal VOC val. (With this small-scale COCO
training it will be **well below** the paper's 72.3 — that's expected. The point is a working,
repeatable metric to improve against.)

### 5b. Scale toward the paper
Once 5a works, close the gap to the paper's numbers by:
- training on **full `train2017`** (and ideally adding GCC/YFCC as the paper does),
- using a **large global batch** (multi-GPU; the code auto-uses all visible GPUs via DataParallel),
  more epochs, and the paper's optimizer (AdamW, lr 5e-4 cosine, weight decay 0.2),
- re-running `evaluate_miou.py` and tracking mIoU upward.

**Success criterion:** mIoU climbs meaningfully with scale; document the numbers in `README.md`.
